### decomper/cluster-prop-ov000

**Goal:** Run the same cluster-propagation pipeline that brief 015
piloted, but against the **#2 remaining cluster**:
`ov000 / func_ov000_021aa4a0` (size 0x14, sig 1, **72 unmatched
siblings**, 9 already matched). Anchor + propagate + ship validated
siblings only.

**Context:**

- Brief 015 piloted the pipeline against cluster #1 (main /
  `func_020498f0`, 148 unmatched). PR #237 shipped **18
  byte-identical matches** out of 144 candidates (107 main + 37
  ov002 — ov004's 4 were brief-non-scope). 89 main + 37 ov002
  candidates dropped at the `TemplateMismatch` gate because their
  actual instruction shape diverged despite sharing the
  `(size, reloc-sig)` fingerprint.
- That **12.5% hit rate** is the new realistic expectation for
  signature-1 cluster propagation. Don't be discouraged if cluster
  #2 yields 5-15 matches not 72; the cluster-finder fingerprint
  isn't unique enough to guarantee bulk match, but every
  TemplateMismatch is the tool *correctly refusing* to ship a
  wrong-C function.
- Anchor `func_ov000_021aa4a0` is **already matched** (lives at
  `src/overlay000/ov000_021aa4a0.c`). It's a 0x14-byte indirect-
  pointer store:

  ```c
  data_ov000_021c73e0->field_144 = x;
  ```

  9 sibling functions also already matched in `src/overlay000/`,
  including `ov000_021aa4b4.c` and `ov000_021aae70.c`.
- The cluster has 72 unmatched siblings — the
  `find_pattern_clusters.py --version eur ov000 0x021aa4a0`
  per-anchor view will list every address. Most are presumably the
  same `ptr->field_NNN = arg` shape with different field offsets
  and different global-pointer-variables.

**Approach (refined from brief 015's lessons):**

1. **Confirm the anchor is matched.** It already is — sanity-check
   `src/overlay000/ov000_021aa4a0.c` exists at 100% in
   `build/eur/report.json` (or `ninja report` if you need to
   regenerate). No new anchor work needed.
2. **Inspect cluster members** via
   `python tools/find_pattern_clusters.py --version eur ov000 0x021aa4a0`.
   Note the `--ranking sig=1` output — these are functions whose
   reloc fingerprint matches the anchor.
3. **Propagate.**
   ```
   python tools/propagate_template.py apply \
     --template src/overlay000/ov000_021aa4a0.c \
     --cluster ov000 0x021aa4a0 \
     --version eur
   ```
4. **Build + report.** `ninja rom`, `ninja objdiff`, `ninja
   report`. Look at the `report.json` deltas for the propagated
   files: drop any that show <100%, keep the rest.
5. **Delinks.** `tools/scaffold_batch.py --apply-delinks` for the
   `complete` blocks (or use `tools/cluster_wave_propagate.py` if
   that one cloud just merged supports the full flow — check its
   `--help`).
6. **Symbol renames.** Each propagated file inherits the anchor's
   shape (struct typedef, field_144 name, the `ov000_obj_144_t`
   typedef). For siblings that store at a *different* offset
   (e.g. `field_148`), the propagation tool should adjust — but
   double-check the generated typedef per file is correct
   (`#define field_<N> field_<N>` style). If the tool produces
   typo'd struct names (`ov000_obj_148_t` etc.), keep them
   consistent.

**Scope:**

- `src/overlay000/<address>.c` — propagated siblings, generated by
  tool, hand-validated only by *dropping* failures.
- `config/eur/arm9/overlays/ov000/delinks.txt` — append-only
  `complete` blocks (one per validated file).

**Non-scope:**

  don't patch.
- ❌ ov004 candidates (failing module).
- ❌ Hand-matching the dropped siblings from brief 015's

- ❌ Renames beyond what the propagation tool emits naturally.
- ❌ `tools/`, `libs/`, `include/`, `AGENTS.md`, `docs/state.md`.
