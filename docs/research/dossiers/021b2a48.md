## Struct context
No global data. All inputs are stack/register arguments. The function packs:

| expression | bits | purpose |
|-----------|------|---------|
| a2 & 0xff | [7:0] of `hi` | palette index / attribute byte |
| 0x4000 | bit 14 of `hi` | mode flag |
| 0x80000000 | bit 31 of `hi` | sign/attr bit |
| (a1 & 0x1ff) << 16 | [24:16] of w[0] | 9-bit X coordinate field |
| a3 \| 0x800 | [10:0] of hh[0] | tile index with bit-11 set |
| a4 << 12 | [15:12] of hh[0] | 4-bit palette/priority |

`hi` is hoisted before the loop because it is constant per call (derived only from `a2` which does not change). The `orr-op-first` recipe means the accumulator register (`hi`, r7) must appear as the FIRST operand in the `orr` instruction: `orr r_,r7,r_shifted`.

The `a5 / 32` signed division: the risk note says the disasm shows `(x + ((x>>4)>>>27)) >> 5` — this is the standard compiler idiom for signed division by a power of 2. Writing `a5 / 32` directly should produce the same magic; do NOT write `a5 >> 5` (that omits the rounding correction for negative dividends).

The stack layout for `w[1]` (int) and `hh[1]` (short) must match the prologue. Declaring `int w[1]; short hh[1];` at function top (before `n`, `i`, `hi`) follows the recipe `decl w,hh,n first`.
