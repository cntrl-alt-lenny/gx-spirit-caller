## Struct context

`data_02104f4c` — system flag array; word at `[1]` (`+0x4`) contains a packed bitfield.
The 3-bit extract `(data_02104f4c[1] << 29) >> 29` is the same `lsl #0x1d / lsr #0x1d`
idiom as seen in ov013. This gives the low 3 bits of the word-1 value — i.e. bits [2:0].

`data_ov006_021cbeb8` and `data_ov006_021cbecc` — kana substring table pointers
(hiragana string blobs from the `021ceb4c` area; Ov004Ov006DataTables.md §Type D).

| p offset | description |
|----------|-------------|
| `p[0]` | kana_id: 3-bit value from `data_02104f4c[1]`, stored as u8 |
| `p + 0x1` | passed to `func_020a978c` as base |
| `p + 0x1c` | passed to `func_02011b9c` as base (twice, with different table args) |
| `p + 0x6298` | guard word: set to 1 at entry |

## Try this

1. The 3-bit field extract `(data_02104f4c[1] << 29) >> 29` in C must be computed on a
   `signed int` to get the `lsl #0x1d / lsr #0x1d` pair. If you use `unsigned int`, the
   compiler may emit `& 7` instead — keep it as signed `int` arithmetic.
2. The second `Fill32(0, p, 0x124)` uses `p` (not `data_ov006_0224f448`) as the
   destination — this zeroes the per-slot template at `p`. Both Fill32 calls must be
   present.
3. `func_0202c0c0` takes the literal `0x486` (SE id); define `OV006_SE_KANA 0x486` as
   a named constant.
4. `func_02011b9c` is called TWICE with different table args: first with
   `data_ov006_021cbeb8`, then (after `func_ov006_021b7f6c` + `func_ov006_021b7e80`)
   with `data_ov006_021cbecc`. The order and interleaving with `7f6c`/`7e80` is
   critical.
5. `func_ov006_021b7f6c` (not `021b7cfc`) — verify you are using the correct function
   pointer; `7f6c` and `7cfc` are different.
6. The main loop here is unguarded (no `slot <= 1` check) — copy directly from the
   `021b6508` unguarded form.
