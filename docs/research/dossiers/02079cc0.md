## Closest matched example
**Path:** src/main/func_0207d430.c
**Key lesson:** The key codegen trick is operand order: `add = add + src[i]` (accumulator first) forces mwcc to emit `add r2, r2, ip` with carry in r2 as both source and destination. Then `add = add >> 16` on a SEPARATE statement (not combined with the store) forces `movs r2, r2, lsr#16` which sets the Z flag for the following `beq tail`. The `movs` (setting flags) is only emitted when `>>16` is the last operation on `add` before the branch.

## Try this
1. Keep `add = add + src[i]` with accumulator (`add`) as the FIRST operand — NOT `src[i] + add`. This maps the accumulator to the first source register of `add rd, rn, rm` (same register for rd and rn), producing `add r2,r2,ip` in objdiff. If operand order is swapped, mwcc may emit `add r2,ip,r2` and a different scheduling window.
2. The `add = add >> 16` statement MUST appear as its own statement after the store — NOT as `dst[i] = (unsigned short)(add >> 16 << 16 ^ add)` or any combined form. It must produce `movs r2,r2,lsr#16` with the S-bit set; if objdiff shows `lsr` without S-bit, add a dummy `if(add==0)` to force the compare.
3. If the index recompute (`mov lr/ip, i, lsl#1`) appears in the wrong position relative to the carry shift, try declaring `int j = i;` inside the do-while and using j for src/dst indexing while keeping i for the loop counter.
