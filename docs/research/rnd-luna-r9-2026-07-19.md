# R&D r9 — GPT-5.6 Luna profile + the "Codex finishes too fast" crack (2026-07-19)

**Method.** A lean 21-agent workflow swarm (6 web-research + 3 diagnosis lanes →
12 adversarial-verify lanes), brain-synthesized. Kept lean deliberately: two
Claude worker agents were live on the shared Claude Max quota, so a big swarm
would collide (the r6/r7 lesson). 0 agent errors. This complements r8 (routing);
r9 answers "what IS Luna, and why do our Codex agents finish in 13 min?"

## Part 1 — what GPT-5.6 Luna actually is (verified)

GA **2026-07-09** as the three-tier GPT-5.6 family: **Sol** (flagship) / **Terra**
(balanced) / **Luna** (cost-efficient "nano" tier). Confirmed specs:

- **Built for speed, by design.** Luna generates **~168 tok/s with ~1.8s TTFT**,
  vs Sol (high) ~52 tok/s / ~11s TTFT — **~3× faster generation, ~6× lower
  latency.** OpenAI positions it for *"fast, cost-efficient, high-volume,
  latency-sensitive tasks such as chat, classification, and lightweight agentic
  workflows"* — explicitly **not** sustained long-horizon coding.
- **Price:** $1.00 / $0.10 cached / $6.00 per 1M (in/cached/out) — **~1/5 Sol's
  input price** ($5/$30). AA cost-per-task: Luna $0.21 vs Sol $1.04.
- **Capability (small delta, large cost gap):** Terminal-Bench 2.1 **84.7%**
  (Sol 88.8, Terra 87.4); SWE-Bench **Pro** **62.7%** (Sol 64.6 — note: OpenAI
  did *not* publish SWE-Bench Verified; both trail Claude Mythos/Fable 5 ~80% on
  Pro); AA Coding-Agent Index **75** (Sol 80). Context **1.05M** / 128k out;
  cutoff 2026-02-16.
- **Its real weakness — long context & long horizon.** MRCR v2 8-needle
  **41.3%** (long-context retrieval collapses); "not recommended for long-context
  recall" and the Sol gap "widens on the hardest reasoning." **This externally
  confirms our standing rule** ([[feedback_opus48_integrity_posture]]): *Luna =
  tool-derived/checkable outputs ONLY, never synthesis.*
- **Effort is an adaptive CEILING, not a floor.** GPT-5.6 "reasons adaptively…
  fewer tokens for simpler tasks, harder on complex." Measured: Luna one-shots a
  well-scoped task in ~1–1.5s at *any* effort (High added ~0.5s over Low on a big
  complexity jump). **Setting Luna to High does NOT make it run longer on
  well-scoped work** — it only raises the *available* budget for tasks that
  genuinely need it. (Effort DOES extend wall-clock on genuinely hard multi-step
  work, ~1.4×/step up the ladder — reserve High/xhigh for those.)

## Part 2 — why our Codex agents finish in ~13 min

Three compounding causes, none of which "run it longer" fixes:

1. **Luna's tier is a fast/short-run tier by design** (Part 1). Short autonomous
   runs are the *intended* behavior, not the model quitting.
2. **Premature "declaring done" (10–30% of scope) is a documented, recurring
   Codex-family behavior** — expected default, not a one-off. OpenAI's own fix is
   persistence language, **but for Luna specifically that prompting is
   cargo-cult** (a low-reasoning tier won't hold extended instructions).
3. **Our briefs over-scope.** The 13-min brief-612 ("apply 150+ renames") is the
   proof: its writeup shows the candidate pipeline returned **0 HIGH / 0 MEDIUM**
   names — there were **not** 150 defensible names to apply. Luna correctly built
   the tool, applied the **4** it could defend (cross-region twins), refused to
   invent, and stopped with honest notes. That was **right behavior on an
   over-scoped brief**, not laziness.

## Part 3 — the crack: stop targeting runtime, decouple throughput from dispatch

**"Make the agent run ~1 hour" is the wrong target (high-confidence).** The real
pain is *dispatch frequency* (you messaging the brain every 13 min). Fast
completion is not a defect.

- **Strategy A — engineer longer single tasks: rejected.** LLM completion time
  isn't reliably controllable; padding briefs to hit a clock is a Goodhart trap.
- **Strategy B — autonomous per-agent QUEUE the agent self-chains: adopt.** And
  Codex supports it **natively**: the `/goal` autonomous loop (plan→edit→run→
  observe→repair→repeat), `--full-auto` (no approval prompts, sandbox stays ON),
  headless `codex exec resume --last` to carry context across tasks, **native
  compaction** (self-summarizes state at the context ceiling — analogous to our
  campaign-state docs), and **no hard time/step cap** (24h+/13M-token single runs
  are documented). Safe here because the **3-region byte gate is the merge
  arbiter** — wrong output can't ship.

### The rollout

1. **NOW — "B-lite" (zero new machinery):** hand each Codex agent a **batch of
   3–5 gate-checked queue items** per kickoff instead of one. Cuts dispatch
   frequency ~4–5× immediately; observe collisions/quality.
2. **NEXT — true self-chaining:** each agent works a **lane-partitioned queue
   file**, claims an item with a **committed marker** (collision guard), does it,
   **pastes the raw gate output** into the commit/log, commits, takes the next —
   pinging the brain only on gate-red, empty-queue, or a checkpoint cadence.
3. **Keep persistence in the HARNESS, not the prompt.** "Do not stop until N" is
   the *most* cargo-cult pattern (frontier agents drop to ~3/9 success at 100
   items). `batch_carve`'s commit-on-pass + red→bisect-park + queue re-injection
   **is** the persistence controller — point the prompt at the queue file and let
   the tool re-inject remaining units.
4. **Two kickoff fixes for Codex specifically:**
   - **Drop "give a plan first / narrate progress each step."** Counterintuitively,
     asking Codex for upfront plans/preambles/status during a rollout *causes*
     premature stops. Keep planning-out-loud for the Opus/Sonnet judgment lanes.
   - **Enumerate the target set.** "150+" is unverifiable; ship a committed
     `targets-bNNN.txt` and define done as set-superset across all 3 regions.

## Part 4 — spin-off deliverable: `tools/scope_gate.py`

The swarm surfaced a real hole: **the merge gate is scope-BLIND.** A thin PR that
renames 40 of 150 targets, or does 1 region and skips 2, passes `gate3`
byte-identically. So `gate3` green ≠ "did the whole brief." Fix = a **sibling
scope gate** the agent runs and pastes, exactly like gate3, turning "fast AND
complete" into one reproducible PASS (and killing the coordination overhead — the
brain audits the pasted PASS instead of eyeballing). Checks it should run,
per brief type:

- **Naming:** `naming_census` real-named delta (head−base, summed 3 regions) ≥
  target; **3-region set-equality** (`comm -3` of per-region new-name lists empty
  — catches "did EUR, faked USA/JPN"); `check_match_invariants` clean (catches
  symbols.txt renamed but extern/src not propagated).
- **Carve/drain:** three independent counts agree — `report.json` complete_units
  delta, new `complete` TUs in delinks diff, new `src/*.c` files.
- **"Build a tool" briefs:** tool file committed + a new `tests/test_<tool>.py`
  passes + a **mutation/negative-control** (feed a known-bad input, require
  nonzero exit — kills the always-returns-OK stub, the fastest shortcut).
- **Meta-fix:** a committed **fixed denominator** (`targets.txt`) makes every
  count falsifiable ("which 150, and prove none were cherry-picked").

Recommended: build `tools/scope_gate.py <manifest>` and require its PASS pasted
in every PR alongside gate3's. This is the highest-leverage next build.
