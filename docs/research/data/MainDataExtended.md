# arm9 Main Extended Data Catalog

Comprehensive catalog of arm9 main `.rodata` and `.data` tables that are **not**
already covered in `DataArrayCatalog.md`. All symbols come from
`src/main/data/` (.s and .c files). Addresses and sizes are taken from the
in-file range comments generated by `tools/cluster_c_pattern3_gen.py` and
confirmed against `config/eur/arm9/delinks.txt`:

```
.rodata  0x020B4320 – 0x020C3BBC  (first byte past .text)
.data    0x020C3BC0 – 0x02102C60
```

The six large tables already cataloged in `DataArrayCatalog.md` are **not**
repeated here:
- `data_020b6ec4` (30,720 B tile/entity layout)
- `data_ov004_02200f18` (22,560 B char map)
- `data_020c95b4` USA (14,764 B card/entity DB)
- `data_020c7b44` (4,924 B card attribute table)
- `data_020c8e84` (2,060 B card display/position)
- `data_020fc160` (1,016 B typed data-pointer array)

The 13-slot fn-ptr table `data_020beb58` is cataloged in
`FunctionPointerTables.md`; a cross-reference entry with stride detail is
included here under the rodata section.

## Rodata tables (0x020B4320 – 0x020C3BBC)

### data_020b4320 — Six 8-byte packed records (rodata cluster)

- **Address:** 0x020B4320
- **Size:** 48 bytes (0x30)
- **Section:** .rodata
- **Element type:** u8 (packed 8-byte struct)
- **Stride:** 8 bytes
- **Count:** 6 records
- **Content:** Six individually exported 8-byte values; each record has two
  significant bytes in positions 0–3 and four trailing zero bytes. The values
  `0x20 0xff 0x09 0x60`, `0x00 0xff 0x01 0xb0`, `0x00 0xff 0x01 0xd0`, etc.
  appear to be packed display-coordinate or MMIO-region descriptors (the
  `0xff` byte is a consistent second element).
- **Indexing:** Cluster-C Pattern-3 generator output (brief 125); 6 separate
  exported symbols at +0, +8, +16, +24, +32, +40.

```asm
data_020b4320: .byte 0x20, 0xff, 0x09, 0x60, 0x00, 0x00, 0x00, 0x00
data_020b4328: .byte 0x00, 0xff, 0x01, 0xb0, 0x00, 0x00, 0x00, 0x00
data_020b4330: .byte 0x00, 0xff, 0x01, 0xd0, 0x00, 0x00, 0x00, 0x00
data_020b4338: .byte 0x20, 0xff, 0x09, 0x60, 0x00, 0x00, 0x00, 0x00
data_020b4340: .byte 0x20, 0xff, 0x11, 0x68, 0x00, 0x00, 0x00, 0x00
data_020b4348: .byte 0x20, 0xff, 0x01, 0xb8, 0x00, 0x00, 0x00, 0x00
```

### data_020b46e0 — 18-entry data-pointer array (rodata)

- **Address:** 0x020B46E0
- **Size:** 72 bytes (0x48)
- **Section:** .rodata
- **Element type:** u32 (data pointer)
- **Stride:** 4 bytes
- **Count:** 18 entries
- **Content:** Flat array of 18 pointers into the `data_020c3c**` family
  (addresses 0x020C3CA8–0x020C3D90, spaced 0x0C apart). These point to
  per-entry records in a larger struct array; the pointer order is shuffled
  (not monotone), suggesting this is a look-up-by-index indirection table
  (e.g., mapping logical slot IDs to struct instances).
- **Indexing:** Pointees are in the `.data` section around 0x020C3CA8.

```asm
data_020b46e0:
    .word data_020c3d00   ; [0]
    .word data_020c3d24   ; [1]
    .word data_020c3d0c   ; [2]
    .word data_020c3d60   ; [3]
    ; ... (18 total)
```

- **Content:** Small integer values: `0x0018, 0x0018, 0x0018, 0x0001, 0x001f,

- **Size:** 24 bytes
- **Element type:** u16

- **Content:** Sequential-ish tile IDs: `0x006d, 0x006e, 0x006f, 0x0070,
  0x0071, 0x0072, 0x0073, 0x0074, 0x0075, 0x0076, 0x0077, 0x0079`. Dense
  range 109–121 (skipping 120=0x78); likely sprite/OAM index array.

#### data_020b47ac — 174-byte sparse u16 index map (lookup table A)

- **Address:** 0x020B47AC
- **Size:** 174 bytes
- **Element type:** u16 (little-endian)
- **Stride:** 2 bytes
- **Count:** 87 entries (many zero/gap entries)
- **Content:** Sparse mapping table; input index (0x00–0x56) maps to output
  index 0x00–0x31; large zero runs indicate holes. Values go up to 0x31=49.
  Classic "keycode → internal ID" or "card type → slot" lookup.

#### data_020b485a — 174-byte sparse u16 index map (lookup table B)

- **Address:** 0x020B485A
- **Size:** 174 bytes
- **Element type:** u16
- **Stride:** 2 bytes
- **Count:** 87 entries
- **Content:** Parallel sparse map; output values range 0x37–0x67; offset
  version of lookup table A above (second half of a two-table pair).

### data_020c4204 — 44-entry structured dispatch table (data)

- **Address:** 0x020C4204
- **Size:** 704 bytes (0x2C0)
- **Section:** .data
- **Element type:** packed struct `{void *fn_or_data, void *data, u32 param1, u32 param2}` = 16 bytes/entry
- **Stride:** 16 bytes
- **Count:** 44 entries
- **Content:** Each entry is:
  - `[+0]` function or data pointer (fn_ptr or data_ptr)
  - `[+4]` associated data block pointer
  - `[+8]` 4-byte parameter word 1 (often `0xFFFFFFFF` = disabled, or overlay ID)
  - `[+12]` 4-byte parameter word 2 (often `0xFFFFFFFF` or sub-index)

  Entries reference functions across many overlays (`ov000`, `ov001`, `ov004`,
  `ov006`, `ov008`, `ov009`, `ov022`) and main functions. Where param1 is
  `0xFFFFFFFF` the entry has no overlay constraint; where it is `0x04`/`0x05`/
  `0x08`/`0x09` etc. it encodes a required overlay ID. This is the **main
  game-mode / scene dispatch table**: indexed by scene/mode ID, each row
  selects the loader, associated data, required overlay, and sub-scene ID.

```asm
; Example entries:
.word func_02001540         ; [0] main-mode fn
.byte 0xff,0xff,0xff,0xff,0xff,0xff,0xff,0xff   ; no overlay constraint
.word data_020c3fb8         ; associated data block
; ---
.word func_ov004_021cb518   ; [2] overlay-4 scene
.byte 0x04,0x00,0x00,0x00, 0x01,0x00,0x00,0x00 ; overlay=4, sub=1
.word data_020c4178
```
