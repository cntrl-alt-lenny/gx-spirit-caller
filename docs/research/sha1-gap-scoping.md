# SHA1-gap scoping — path to byte-identical ROM rebuild

**Brief:** 137 (cloud scoping). After brief 134 closed the
27/27-module gap on EUR (26/27 on USA / JPN), `ninja sha1` still
fails — bytes outside dsd's per-module checksum coverage diverge
from orig. Brief 137 inventories the ROM-level diff and outputs
the brief 138+ application plan.

## TL;DR

**99.995% of the SHA1 gap is one root cause: `.DS_Store` macOS
metadata files leaking into the ROM file image.** Deleting them
collapses EUR's ROM diff from **100,805 bytes → 5 bytes**.

The 5 remaining bytes (EUR) break down as:

- **4 bytes**: two CRC16 fields in the ROM header (`0x6c` logo
  CRC, `0x15e` header CRC) that dsd's `rom build` doesn't compute
  — leaves the logo CRC zeroed and propagates an incorrect
  header CRC.
- **1 byte**: ov4's `ram_size` field in the overlay table is
  shifted by 0x400 because brief 134's veneer patcher has an
  off-by-1024 in its idempotent "already-patched" code path
  (re-derives the YAML `code_size` field as
  `len(data) - 1024` even when `data` is the post-patch size).

USA / JPN have **7 bytes diff** (the same 5 + 2 pre-existing
function-displacement bytes in main that were flagged
out-of-scope in brief 134).

All three categories are FIXABLE; total work is small (3
narrowly-scoped briefs). The `.DS_Store` fix alone takes EUR /
USA / JPN from 100K+ bytes to single-digit byte-diffs — a
3-orders-of-magnitude reduction unlocked by a one-line build
filter.

| USA    | 100,782 | 7 | 1 | 84,103 | 16,651 | 2 | 18 |

Same shape across all three regions. ARM9 program diffs (USA/JPN)

### Cat ε dominance — the FNT + FAT explosion

84,119 FNT + 16,661 FAT + 18 spillover = **100,798 bytes** —
essentially the entire gap. Inspecting the FNT sub-tables:

```
orig  first dir entry: 88 41 63 61 64 65 6d 69 61   (".Academia", len=8)
built first dir entry: 09 2e 44 53 5f 53 74 6f 72 65  (".DS_Store", len=9)
                       └─ extra entry inserted BEFORE every real file
```

dsd's file scanner picks up `.DS_Store` from
`extract/<region>/files/.DS_Store` and slots it into the FNT
ahead of `Academia` (sort order). Every downstream FNT entry
shifts by 10 bytes; every FAT entry shifts by 8 bytes. The
header's `fnt_size` (`+10`) and `fat_size` (`+8`) fields
auto-derive from the bigger tables. The "spillover" 18 bytes
are the FNT / FAT tail extending past orig's table-end into the
alignment pad region.

`.DS_Store` is auto-generated by macOS Finder and is not
tracked in git (the project's `.gitignore` covers
`extract/`). It's an environment-specific build artifact that
nothing should have included.

## Categorization

### Cat α — ROM header CRC16 fields (4 bytes)

Two CRC16 fields in the NDS header are wrong in built ROMs of
all three regions.

| Offset | Field | orig (LE) | built (LE) | Spec |
|--------|-------|----------:|-----------:|------|
| `0x6C` | Nintendo logo CRC16 | `0x6FF7` | `0x0000` | CRC16 over header bytes `0xC0..0x15C` (the logo image) |
| `0x15E` | Header CRC16 | `0x15E5` | (varies) | CRC16 over header bytes `0x00..0x15D` |

dsd's `rom build` leaves the logo CRC zeroed. The header CRC is
computed but uses the (zero) logo CRC as input, so it differs
from orig.

**Fix shape**: small tool / dsd PR. Both CRC algorithms are
public (NDS header spec). Trivial Python implementation. Wire
into the build chain as a post-`dsd rom build` step.

**Workflow wall risk**: low. CRC computation is well-defined.

### Cat β — Overlay table entry (1 byte)

Overlay 4's `ram_size` field at ROM file offset `0x8b289` differs
by 1 byte (0x17 → 0x13). This is the upper byte of a 4-byte
field that ultimately stores 0x000417a0 (orig) vs 0x000413a0
(built) — a difference of 0x400 = 1024 bytes.

**Root cause**: brief 134's `tools/patch_ov004_veneers.py` has
an off-by-1024 bug in its idempotent "already-patched" code
path. In the `rom_config` rule the patcher runs a second time
(after `dsd rom config` regenerates the YAML). On this second
invocation `data` is the already-spliced binary (268,192 bytes
for EUR), so:

```python
expected_output_size = len(data) - EXPECTED_OUTPUT_DELTA
#                    = 268192 - 1024
#                    = 267168   # WRONG — should be 268192
```

The 267168 value gets written into `arm9_overlays.yaml`'s ov4
`code_size`. `dsd rom build` then aligns that to the next 512
boundary (267680 = 0x413a0) and writes it into the ROM overlay
table — 512 bytes short of orig.

**Fix shape**: one-line patcher change. Compute
`expected_output_size = len(data)` when the binary is
already-patched, `len(data) - EXPECTED_OUTPUT_DELTA` only when
the splice will actually run.

**Workflow wall risk**: zero. Pure tool bug, fully understood,
fixed in <10 lines.

### Cat γ — Banner / title metadata (0 bytes)

No diffs. The banner region is byte-identical across all 3
regions. Title/icon strings unchanged.

### Cat δ — Padding regions (0 bytes)

No diffs that aren't already attributed to Cat α/β/ε. The 18
"spillover" bytes (Cat ε) extend the FNT/FAT into orig's
alignment pad area, but those are caused by the `.DS_Store`
extra-entry pressure, not padding-corruption.

### Cat ε — Unexpected: `.DS_Store` macOS junk (99.995% of pre-fix gap; 0 bytes post-fix)

`.DS_Store` macOS metadata file inclusion. Detailed above.

**Fix shape**: filter the file out before `dsd rom build`
scans `extract/<region>/files/`. Two options:

1. **Build-time pre-clean** (recommended): add a tiny tool
   `tools/clean_macos_junk.py` that runs before `dsd rom build`
   and deletes any `.DS_Store`, `._*`, `Thumbs.db`, or similar
   OS-generated junk under `extract/<region>/files/`. Wire it
   into the ninja graph as a phony rule prerequisite of `rom`.
2. **Upstream dsd filter**: file an issue on
   [`ds-decomp`](https://github.com/AetiasHax/ds-decomp) asking
   `dsd rom build` to skip well-known OS-junk filenames during
   file image scanning. Slower to land but the cleanest fix.

Recommend doing **both**: ship (1) immediately, file (2)
upstream as the long-term fix and remove the local tool when
dsd adopts the filter.

**Workflow wall risk**: low.

### Cat ζ (NEW — USA / JPN only): pre-existing main function-displacement (2 bytes)

Not part of brief 137's responsibility — flagged in brief 134.
The 2-byte BL-offset diffs at file `0x89ac0` / `0x89b10` are a
function-tier decomp gap (function in USA/JPN main is at the
wrong VA). Resolves when more source-level matching happens in
USA/JPN main. Brief 135+ candidate.

## Brief 138+ application plan

### Brief 138 — `.DS_Store` filter (cloud, HIGH)

**The 99.995% fix.** Add `tools/clean_macos_junk.py` (or
similar) that recursively deletes `.DS_Store`, `._*`,
`Thumbs.db`, `desktop.ini` under `extract/<region>/files/`
before `dsd rom build` runs. Wire as a phony prerequisite of
the `rom` target — runs once per ninja invocation, idempotent,
silent when nothing to delete.

**Verification**: post-PR `ninja rom && cmp -l ...` shows
EUR drops to 5 bytes / USA + JPN to 7 bytes.

**Optionally**: file upstream issue on
[`ds-decomp`](https://github.com/AetiasHax/ds-decomp) for a
permanent in-tool filter.

**Workflow wall**: none new. Ships as a sibling of brief 131's
`patch_lcf_arm9_align.py` integration pattern.

### Brief 139 — Brief 134 patcher off-by-1024 fix (cloud, MEDIUM)

**Root cause**: `tools/patch_ov004_veneers.py`'s
`expected_output_size` is computed before the already-patched
check, so on the YAML-only second invocation it derives a
1024-bytes-too-small value. Fix is a 2-line change plus a
regression test.

**Verification**: post-PR `cmp` shows the `0x8b289` byte gone
across all 3 regions; ov4 overlay table ram_size matches orig.

**Workflow wall**: none.

### Brief 140 — ROM header CRC16 computation (cloud, MEDIUM)

**The "last 4 bytes" fix.** Two CRC16 fields:

- Logo CRC at `0x6C`: CRC16 over header bytes `0xC0..0x15C` (the
  Nintendo logo image, 156 bytes).
- Header CRC at `0x15E`: CRC16 over header bytes `0x00..0x15D`.

Both use the standard NDS CRC16 polynomial (`x^16 + x^15 + x^2
+ 1`, init 0xFFFF — see GBATEK / NDS header docs). Implement
in a Python tool, wire as a post-`dsd rom build` patcher
(modeled on brief 131 + brief 134 patcher chain pattern).

**Verification**: after this brief, EUR drops from 5 → 0 bytes
diff → SHA1 match achieved.

**Workflow wall**: W8 candidate (NEW) if upstream dsd should
own this — file an issue. The CRCs are a STANDARD output of any
NDS ROM builder; dsd not computing them is arguably a bug.
Local tool ships immediately; upstream fix follows.

### Optional brief 141 — USA / JPN main 2-byte residue

Function-tier decomp gap, flagged in brief 134. Resolves
when the 244-byte-displaced function gets source-matched in
USA / JPN main. Decomper work; not blocking the 27/27 +
SHA1-EUR-match milestone.
