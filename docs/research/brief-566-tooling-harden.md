## Fix 1 (S1) — in-loop C compile+objdiff feedback

### Before

```
$ echo '{"tool_name":"Edit","tool_input":{"file_path":".../src/main/func_X.c",...}}' \
    | python .claude/hooks/post_edit.py
$ echo $?
0
```

Silent pass, no output — `post_edit.py`'s very first content check
(`if rel.suffix != ".py": return 0`) short-circuits before anything
else runs. Editing a game `.c` file gives the agent nothing; it has to
remember `ninja` + `ninja objdiff` by hand, exactly as the swarm's S1
finding describes.

### The fix

Extended `.claude/hooks/post_edit.py` (and its separately-registered
`.codex/hooks/post_edit.py` twin — both hook JSONs point at their own
copy, so both needed the identical change) to handle
`src/**/*.{c,cpp,s}` edits:

1. **Detect the active region** by reading `build.ninja` for the first
   `game_version = (eur|usa|jpn)` marker — `build.ninja` is single-
   region at a time (each `configure.py <ver>` run overwrites it), so
   this reliably tells us which region is currently configured.
2. **Skip region-mismatched edits** silently — e.g. editing
   `src/usa/main/func_X.c` while `build.ninja` is EUR-configured has
   no valid ninja target, so the hook no-ops rather than erroring.
3. **Compile the single TU**: `ninja build/<region>/<path-minus-ext>.o`
   (works identically for `.c`, `.cpp`, and `.s` — `dsd`'s per-TU
   build rules follow the same `build/<region>/<src-path>.o` naming
   convention regardless of source language, just via a different
   ninja rule, `mwcc` vs `mwasm`, internally).
4. **On a clean compile, run a single-unit `objdiff-cli` diff** and
   report the match percentage.
5. **On a failed compile, surface the real compiler error** (mwccarm's
   actual stderr, including line numbers) instead of nothing.

Both steps use hard subprocess timeouts (8s compile / 5s objdiff) as a
safety net — the brief's target is "~1-3s," which is what a real leaf
`.o` compile takes; the timeouts only matter if something hangs.

**A tooling gotcha found and worked around**: `objdiff-cli diff -p .`
(project mode) requires a pre-built `objdiff.json`, which is only
generated by the slow, whole-project `ninja objdiff` (`dsd objdiff
--scratch` across every unit in the game) — using project mode would
have blown the hook's latency budget by orders of magnitude. Switched
to `objdiff-cli diff -1 <target.o> -2 <base.o>` (direct two-file mode),
which needs no project config and only touches the two `.o` files for
the single edited unit — `build/<region>/delinks/<unit>.o` (the
delinked original) vs `build/<region>/<unit>.o` (the freshly compiled
one).

## Fix 2 (A3) — the C% metric's 99.907% ceiling

### The bug

`c_code_bytes()` (the "true readable C" numerator, brief 561) summed
only `.text` bytes for `.c`/`.cpp`-sourced TUs. But the denominator
used everywhere (`total_code`, from `CODE_SECTIONS = {".text",
".init"}`) sums `.text` **and** `.init`. Since `c_code_bytes()`'s scan
never looks at `.init` ranges — even for a `.c` TU that owns some
(`sinit_*.c` static-initializer glue) — those bytes can never enter
the numerator, so the ratio is structurally capped at
`1 - init_bytes/total_code`, confirmed here at **99.9069%** for EUR
(matches the swarm's independently-cited 99.907% almost exactly).

A second, smaller bug rode along: the old numerator scan used a
hand-rolled regex pair (`_C_SRC_HDR` / `_C_TEXT_RANGE`) that tracked
"current TU's source extension" as a variable persisting across the
whole file scan. A `_dsd_gap@...` TU header never matches the
`src/`or`libs/`-prefixed regex, so it never reset that variable — a
gap TU's `.text` bytes immediately following a real `.c` TU would be
silently misattributed to C. (Measured zero live impact on this
repo's current EUR tree — no `.c` TU happens to be immediately
followed by a gap TU right now — but it's a real correctness bug that
could bite as `delinks.txt` evolves, and the regression tests pin it
either way.)

### The fix

- **`c_code_bytes()`** now uses the canonical `parse_delinks_file`
  (already used by `summarize_delinks`) instead of the bespoke regex
  scan — each TU's `source` field is correctly scoped per-TU by the
  parser, eliminating the ext-leak by construction. Added a
  `require_complete: bool = False` parameter for the stricter
  "verified byte-matching only" view (gates on `status == "complete"`,
  matching what `summarize_delinks`'s `matched_code` requires); the
  default stays ungated, preserving the existing, deliberate
  "an in-progress `.c` draft is still real C on disk" semantics an
  existing test (`test_incomplete_c_tu_still_counted`) already pins.
- **New `c_code_total_bytes()`** — sums `.text`-only from the
  module-level section map (same source as `total_code`, minus
  `.init`). This is the correct denominator to pair with
  `c_code_bytes()`'s numerator, since both now count exactly the same
  section set.
- Updated `progress.py`'s `print_report()` and both `--json` payload
  paths (report.json tier and delinks.txt tier), plus
  `generate_progress_bars.py`'s `region_metrics()`, to use
  `c_code_total_bytes()` for the C%-specific ratio. The unrelated
  "code"/"matched" line (byte-matched including `.s` ships) still
  correctly uses the `.text`+`.init` `total_code` — that metric's
  numerator also spans both sections, so no mismatch there; only the
  C%-specific pairing needed realigning.

### The corrected number

```
OLD (buggy):  191556 / 2388172  = 8.0210%
NEW (fixed):  191556 / 2385948  = 8.0285%
denominator delta (.init dropped): 2224 bytes
```

The numerator is unchanged (0 delta) — confirming the ext-leak fix has
zero effect on today's actual number (nothing in the current EUR
`delinks.txt` tree triggers it) while still being a real, tested
correctness fix. All figures computed live against this worktree's
`config/eur` via the delinks.txt tier (no build required, matching how
CI would see it); a stale `build/eur/report.json` left over from an
earlier session was deleted first — mixing it into the comparison
briefly produced a nonsensical result (my new `.text`-only denominator
appeared *larger* than the old `.text`+`.init` one) because report.json
was frozen days-stale relative to the current delinks.txt tree, not
because of a bug in the fix; deleting it and using the always-live
delinks.txt-sourced tier resolved the apparent paradox immediately.
