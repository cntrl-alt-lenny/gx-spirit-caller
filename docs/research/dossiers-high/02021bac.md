## SHA1 pitfall

**Primary mode:** Mode A — pool-word order; dual-symbol aliasing of the same underlying address; both wildcarded by verify.py, only ninja report catches a symbol collapse

**Why this recipe hits this mode:** Both `data_02197434` and `data_02198434` are named globals used as load bases for the same physical address viewed through two extern symbols. Verify.py wildcards both relocations. If the compiler collapses them to one pool word (one symbol for both accesses) the pool count changes silently — only `ninja report` exposes the mismatch.

**Pool words to verify against orig .s:**
- `data_02197434` — base for the entry array (`+0x1048`), stride pointer through the loop, and in-loop count reload path (`+0x10b0` via r4 live base)
- `data_02198434` — pre-loop guard: count read at `+0xb0` (same physical address as `data_02197434+0x10b0` but via a different symbol)

**Verify gate sequence:**
1. verify.py src/main/func_02021bac.c func_02021bac [--cc all] — instruction bytes
2. ninja report — resolves pool words; catches Mode A dual-symbol collapse before sha1
3. ninja sha1 — final gate
