# func_02021660 (main, HIGH class A, 80B)

**Recipe:** Search loop: running-pointer access + separate bound index
**Risk:** reg-alloc of matched/index/pointer may differ; mwcc could index e[i] instead of incrementing e, or reorder the add lr,#0xc
**Confidence:** med

## SHA1 pitfall

**Primary mode:** Mode E — visible instruction byte diff; pointer-vs-index codegen and reg-alloc split are directly visible to verify.py

**Why this recipe hits this mode:** The reg-alloc split between the running pointer `e` (expected in lr) and the separate loop index `i` (expected in r3) is compiler-sensitive. If mwcc chooses `(int*)(p+0x18) + i*3` array indexing over pointer increment, or places the `add lr,#0xc` in a different position within the loop, the instruction stream differs visibly and verify.py reports it immediately.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (compared directly by verify.py).

**Verify gate sequence:**
1. verify.py src/main/func_02021660.c func_02021660 [--cc all] — instruction diff is visible directly
2. ninja sha1 — final gate

## Reshape hints

- Declare `e` as `int *e = (int *)(p + 0x18)` and advance with `e += 3` at the bottom of the loop (not `e = (int*)(p+0x18) + i*3`) to coax the lr running-pointer pattern
- Keep `matched`, `i`, and `e` as three separate named locals so mwcc allocates them to distinct registers matching the orig's lr/r3 split
- The stride increment `e += 3` (== +0xc bytes) must come at the end of the loop body after all accesses to `e[0]` and `e[1]`, to match `add lr,#0xc` placement in the orig .s
- If objdiff shows `e` in a different register than lr, try declaring it before the for-init slot (move `int *e = ...` above the for loop)
- For the early-return `return e[1]`, use the live pointer directly (not a reload) so mwcc does not insert an extra LDR between the key-match and the value load
