## SHA1 pitfall

**Primary mode:** Mode A — wrong pool-word value (HIDDEN; only ninja report catches)
**Secondary mode:** Mode E — visible instruction diff (ands operand order)

**Why this recipe hits this mode:** `data_0219daec` is a named global used as the null-guard load base and the +0x644 field base; both loads come from the literal pool and verify.py wildcards them. A wrong symbol address silently passes verify.py. Additionally, the ANDS instruction requires the field be the first operand (`field & (1 << key)` not `(1 << key) & field`) to match the `ands r0, r3, lsl r...` encoding — writing it backwards produces a visible byte diff.

**Pool words to verify against orig .s:**
