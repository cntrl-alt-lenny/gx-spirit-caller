# func_0202ab04 (main, HIGH class A, 96B)

**Recipe:** counted loop, signed-guard call+clear, bit-clear tail
**Risk:** the separate +0xa7 induction pointer (asm keeps r5) may instead recompute base+i*0x10; strh/-1 ok

## SHA1 pitfall

**Primary mode:** Mode E - Visible instruction byte diff (verify.py catches directly)

**Secondary mode:** Mode B — callee `func_02022580` address wildcarded by verify.py BL reloc; only ninja report catches a wrong value

**Why this recipe hits this mode:** The assembly holds a dedicated r5 pointer for the `+0xa7` byte offset within each stride-0x10 element; if mwcc recomputes `b + (i<<4) + 0xa7` on each iteration rather than walking a persistent pointer, the instruction sequence changes visibly. The `strh`/-1 half-word store and the `ldrb`/`strb` bit-clear byte are also visible in the diff.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (compared directly by verify.py).

**Verify gate:**
1. python tools/verify.py src/main/func_0202ab04.c func_0202ab04 --cc all (catches diff directly)
2. ninja report (resolves BL reloc for func_02022580; catches Mode B mismatch)
3. ninja sha1 (final gate)

## Reshape hints

- Match loop induction to .s: if r5 stays as a persistent pointer to `b + (i<<4) + 0xa4` across iterations, declare `short *p` outside the inner body and increment `p` by `0x10/sizeof(short)` each iteration
- The `*p = -1` store uses `strh`; declare the field as `short` not `int` to ensure mwcc emits `strh` not `str`
- The `b[(i<<4) + 0xa7] &= ~1` bit-clear must use `ldrb`/`strb`; declare `char *b` for the base pointer
- Keep loop body in exact textual order: signed-guard, call, clear (`*p = -1`), bit-clear byte — reordering changes the visible instruction sequence
- Cross-check `func_02022580` BL offset in orig .s against its expected address via ninja report before sha1
