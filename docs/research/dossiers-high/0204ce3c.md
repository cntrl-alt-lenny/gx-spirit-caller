## SHA1 pitfall

**Primary mode:** Mode B — indirect callees wildcarded by verify.py BL reloc

**Why this recipe hits this mode:** `func_020498f0` is an accessor BL'd three times and `func_0204ead0` is an extern BL target — verify.py wildcards all BL relocations, so a wrong callee address at either site passes instruction-byte checks silently and only ninja report resolves the actual callee bindings.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (compared directly by verify.py).

**Verify gate sequence:**
1. verify.py src/main/func_0204ce3c.c func_0204ce3c [--cc all] — instruction bytes
2. ninja report — resolves BL targets for func_020498f0 and func_0204ead0; catches Mode B callee mismatches before sha1
3. ninja sha1 — final gate

## Reshape hints

- Hold the first accessor result in a local `a` before calling the accessor a second time; the asm keeps the first result in r4 across the second call, so reloading after the second call would change the register schedule.
- The halfword read of `fa4` must use `unsigned short` (ldrh), not `int` (ldr) — the field is typed u16 at +0xa4 so a signed or word read changes the load instruction visibly.
- Pass the last two arguments (0, 0) as literal int stack arguments to `func_0204ead0` — they become the 5th and 6th parameters and must not be folded or hoisted.
- Write `func_020498f0()->f204 = 0` after the call and before the return; the asm zero-clears the field then returns `r` in r0 — reversing order or caching the pointer changes the store sequence.
- Cross-check addresses of `func_020498f0` and `func_0204ead0` against the .s BL instructions to confirm no callee swap before finalising.
