**Recipe:** setup call + COMMUTATIVE add (accumulator-operand-first) counted loop
**Risk:** stride-8 element modeled as {int;int}; count*8 must lower to lsl#3; unsigned bcc/bls guard

## SHA1 pitfall

**Primary mode:** Mode E — visible instruction byte diff; verify.py catches commutative-add operand order and lsl#3 vs multiply

**Why this recipe hits this mode:** The commutative `add r0,r5,r0` requires `addend + base[i].v` operand order in C; swapping to `base[i].v + addend` flips the register allocation and changes bytes. Additionally `count * 8` must lower to `lsl #3` (unsigned multiply by constant), which requires the operand to be `unsigned`.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (compared directly by verify.py).

1. verify.py src/main/func_0200b1bc.c func_0200b1bc — instruction diff is visible directly (add operand order, lsl encoding)
2. ninja sha1 — final gate

## Reshape hints

- Write the add as `addend + base[i].v` (addend-first) to match the `add r0,r5,r0` commutative form the compiler emits
- The loop counter `i` must be `unsigned` so `count * 8` lowers to a `lsl #3` shift instruction, not a `mul`
- The loop guard must be `i < count` with `unsigned count` so the compiler emits `bcc`/`bls` rather than `blt` (signed)
- Model the stride-8 element as a struct `{int v; int hi;}` so the field access at `+0` doesn't require an extra offset calculation
- If objdiff shows a multiply instead of shift, double-check that `count` is declared `unsigned` all the way through — a signed intermediate will block lsl folding
