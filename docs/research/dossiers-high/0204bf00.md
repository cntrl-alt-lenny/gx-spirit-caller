## SHA1 pitfall

**Why this recipe hits this mode:** `func_020498f0` is called twice via BL for the two byte-comparison guards, and `func_0204bd8c` and `func_0204bf44` are BL targets in the tail; verify.py wildcards all BL relocs so a wrong address for any callee is silent until `ninja report`.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (compared directly by verify.py).

**Verify gate sequence:**
1. verify.py src/main/func_0204bf00.c func_0204bf00 [--cc all] — instruction bytes
2. ninja report — resolves BL relocs; catches Mode B callee-address mismatches before sha1
3. ninja sha1 — final gate

## Reshape hints

- Reload `func_020498f0()->f15` for each guard rather than caching it; the .s calls the accessor twice, once for the `== 2` check and once for the `== 3` check, each via a fresh BL
- The field offset f15 (0x15) is guessed; cross-check the `ldrb` offset in the orig .s and adjust the struct pad if it differs
- `func_0204bf44(1)` must be a plain call, not a tail-call return; the .s epilogue follows it, so write it as a call before the implicit `return` rather than as a tail expression
- The two early-return guards must use `== 2` and `== 3` in that exact order; swapping them changes the comparison constants in the cmp instructions and is visible in verify.py
- Cross-check BL addresses for `func_020498f0`, `func_0204bd8c`, and `func_0204bf44` in the orig .s before running `ninja sha1`
