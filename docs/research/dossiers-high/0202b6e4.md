# func_0202b6e4 (main, HIGH class A, 104B)

**Recipe:** counted scan/insert loop, early-return guards, single call
**Risk:** loop reg-alloc (id in r4, idx in r5) may shift; cmp operand order target-first preserved

## SHA1 pitfall

**Primary mode:** Mode E - Visible instruction byte diff (verify.py catches directly)

**Secondary mode:** Mode B — callee `func_0202b6cc` address wildcarded by verify.py BL reloc; only ninja report catches a wrong value

**Why this recipe hits this mode:** The assembly assigns `id` to r4 and `idx` to r5; swapping these (or spilling one) produces a visible register-allocation diff. The `cmp` of `target == func_0202b6cc(id)` must have target as the first cmp operand to match the .s `cmp r0,target` form, and a reversal is visible directly.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (compared directly by verify.py).

**Verify gate:**
1. python tools/verify.py src/main/func_0202b6e4.c func_0202b6e4 --cc all (catches diff directly)
2. ninja report (resolves BL reloc for func_0202b6cc; catches Mode B mismatch)
3. ninja sha1 (final gate)

## Reshape hints

- Declare `id` before `idx` in the function body to bias mwcc toward assigning r4 to `id` and r5 to `idx`, matching orig register layout
- Write the comparison as `target == func_0202b6cc(id)` (target-first) to match the .s `cmp r0,target` operand order
- The capacity overflow check `if (idx == count) return -1` must come before `out[idx++] = id` — keep this order to match the orig branch sequence
- The null-terminator tail `if (idx < count) out[idx] = 0` must be a conditional store, not unconditional; match the branching form in the .s
- Cross-check `func_0202b6cc` BL offset in orig .s against its expected address via ninja report before sha1
