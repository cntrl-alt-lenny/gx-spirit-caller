## SHA1 pitfall

**Primary mode:** Mode E — visible instruction byte diff; the baserom emits a dead `cmp r1,#7` after setting r0=1 that no C boolean expression reproduces

**Why this recipe hits this mode:** The baserom's sequence is `cmp r0,#2; mov r0,#1; bxne lr; cmp r1,#7; bx lr` — the second `cmp` is dead (it sets flags but r0=1 is already committed). mwcc compiling `a != 2 || b == 7` (or any equivalent) will materialize the `b==7` sub-expression into r0 via `moveq`/`movne`, producing a different instruction sequence. The condition-flag return shape (leaving the result in the Z flag from the last cmp) is not reachable from a standard `int` return.

- The natural source `return a != 2 || b == 7;` is already in the prep file — try it first; mwcc may happen to emit the early-return + dead-cmp shape under specific optimization flags
