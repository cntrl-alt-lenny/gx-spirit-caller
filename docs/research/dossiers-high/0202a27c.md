# func_0202a27c (main, HIGH class A, 124B)

**Recipe:** scan-for-empty loop + Fill32 + stride-mul alloc, dual-pool
**Risk:** loop induction (ptr vs idx*0xec) and r0=#0 fall-through into Fill32 value-arg may shift; pool dedupe risk

## SHA1 pitfall

**Primary mode:** Mode D - .s-ONLY candidate (pool dedup / patcher trim bypass)

WARNING: .s-only candidate. Do NOT write a .c.

**Why this recipe hits this mode:** Like 0202a240, this function relies on `data_0219a934` and `data_0219a934_alias` at the same raw address appearing as two distinct `.word` pool entries. If mwcc deduplicates them the pool collapses. Additionally, the loop induction form (pointer walk vs index recompute) and the `Fill32` value-arg fall-through may independently produce instruction-byte diffs.

**Pool words to verify against orig .s:**
- `data_0219a934` — count-read base; first `.word` pool entry; used as `(int*)base + 4` for element count
- `data_0219a934_alias` — element-base load; second `.word` pool entry (must be distinct, same raw addr) for the `Slot**` LDR

**Verify gate:**
1. python tools/verify.py src/main/func_0202a27c.c func_0202a27c --cc all (instruction bytes)
2. ninja report (resolves pool/BL words; catches Mode A/D mismatch)

## Reshape hints

- Confirm two distinct `.word` entries in raw .s output before spending time on loop tuning; if mwcc deduplicates, escalate to .s-only
- Match loop induction to .s: if orig uses pointer walk (`e++`) rather than index recompute, keep `Slot0202a27c *e` and increment rather than computing `base + i`
- Write `count` load first (via `data_0219a934 + 4`) and `base` load second (via `data_0219a934_alias`) to match pool emission order
- The `Fill32(0, &base[idx], 0xec)` value arg `r0=#0` must not be stale from a prior expression; verify the register is freshly loaded
- For the `if (count > 0)` guard, match whether the .s uses a branch-not-taken skip or a conditional loop entry to avoid a cmp-order Mode E diff
