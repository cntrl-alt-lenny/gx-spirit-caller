# func_0204f310 (main, HIGH class A, 176B)

**Recipe:** init call + setter sequence (store-order) + 5 registration calls
**Risk:** many offsets (f15/f16/f464/f468/f17d/f2d0); acc reloaded per store
**Confidence:** med

## SHA1 pitfall

**Primary mode:** Mode A — named data_ pool globals wildcarded by verify.py, only ninja report catches mismatches

**Secondary mode:** Mode B — BL to func_0204f3c0 and func_02067850 wildcarded by verify.py reloc

**Why this recipe hits this mode:** Five named `data_020ffXXX` arrays are passed by pointer to five `func_02067850` registration calls; each address lands in the literal pool and is wildcarded by verify.py, making wrong addresses invisible in the instruction-byte diff until ninja report resolves them against the orig .s pool.

**Pool words to verify against orig .s:**
- `data_020ff974` — registration pointer passed to func_02067850(0x36, ...)
- `data_020ff980` — registration pointer passed to func_02067850(0x32, ...)
- `data_020ff9a0` — registration pointer passed to func_02067850(0x33, ...)
- `data_020ff9ac` — registration pointer passed to func_02067850(0x34, ...)
- `data_020ff9b8` — registration pointer passed to func_02067850(0x35, ...)

**Verify gate sequence:**
1. verify.py src/main/func_0204f310.c func_0204f310 [--cc all] — instruction bytes
2. ninja report — resolves pool words for all five data_ symbols and BL targets; catches Mode A/B mismatches before sha1
3. ninja sha1 — final gate

## Reshape hints

- The accessor `func_020498f0()` must be reloaded before every field store — each write in the asm reloads via BL; caching the pointer across stores changes the instruction sequence.
- The byte-field store order must exactly match: f15=a0, f16=a1, f464=a2, f468=a3, f17d=0, f2d0=0 — any reorder of byte stores relative to word stores will shift register assignments.
- The five registration calls must appear in id order (0x32 through 0x36); each call uses a distinct data_ pointer that must come from the literal pool, so do not hoist or share pointers.
- Cross-check each data_ symbol address against the matching .s pool word in the order they appear in the literal pool (compilation order, not necessarily id order).
- Cross-check `func_0204f3c0` (called with arg 0 at the top) and `func_02067850` addresses against the respective .s BL instructions.
