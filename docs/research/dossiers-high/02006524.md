## SHA1 pitfall

**Primary mode:** Mode A — hidden pool-word order mismatch; `data_02103da4` and `data_027e0000` are named globals whose pool-word addresses are wildcarded by `verify.py` BL reloc handling but checked only by `ninja report`

**Secondary mode:** Mode B — the three `h()` calls are indirect BLX through a function-pointer field; the callee address is not a named extern but a field load, so the callee identity is implicitly tied to the field offset (0x314/0x310/0x30c); a wrong offset produces a wrong BLX target that `verify.py` wildcards

**Why this recipe hits this mode:** Both `data_02103da4` and `data_027e0000` appear as pool literals; their values are wildcarded by `verify.py`'s reloc pass. Additionally the indirect `h()` invocations load function pointers from guessed field offsets — a wrong offset gives a wrong callee with no direct diff visible in `verify.py` output, only in `ninja report` symbol resolution.

**Pool words to verify against orig .s:**
- `data_02103da4` — struct base for all field accesses (+0x31c, +0x334, +0x314, +0x310, +0x30c, +0x0); reloaded before each indirect call in orig (do NOT bind to a local — the comment warns this diverges)
