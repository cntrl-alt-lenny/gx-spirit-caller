# func_0204b370 (main, HIGH class A, 96B)

**Recipe:** table-stride loop (entry->f4 guard + call) + Fill32 tail
**Risk:** stride struct size 0xc / f4 offset; loop bound 0x9a; spurious sub sp,#4
**Confidence:** med

## SHA1 pitfall

**Primary mode:** Mode A — Wrong pool word value (hidden from verify.py; only ninja report catches it)

**Why this recipe hits this mode:** `data_0219dcb4` is the array base used in every loop iteration and again in the Fill32 tail call; its address is emitted as a pool .word, wildcarded by verify.py, so a wrong global address is invisible until `ninja report`.

**Pool words to verify against orig .s:**
- `data_0219dcb4` — array base for all stride-indexed accesses and the Fill32 destination; the pool .word entry must match the orig .s exactly

**Verify gate sequence:**
1. verify.py src/main/func_0204b370.c func_0204b370 [--cc all] — instruction bytes
2. ninja report — resolves pool words; catches Mode A mismatches before sha1
3. ninja sha1 — final gate

## Reshape hints

- Keep the struct stride at 0xc bytes (Entry0204b370 with two 4-byte pads around f4) so `entry[i].f4` compiles to the correct `ldr rN,[base, i*0xc + 4]` offset pattern
- If `sub sp,#4` appears spuriously in the prologue, the function is being routed through a frame-building path; check whether the loop or Fill32 tail forces a callee-save that requires a stack frame and adjust the source to eliminate it
- The Fill32 tail call uses the same `data_0219dcb4` base and size 0x738 (= 0x9a * 0xc); write these as literal constants to avoid accidental recalculation
- Loop bound 0x9a must be expressed as a literal integer, not derived from a sizeof, to avoid MWCC generating a different constant-propagation path
- Verify that `func_0204548c(4, entry[i].f4, 0)` keeps the constant args 4 and 0 as immediate literals and does not spill them to the stack
