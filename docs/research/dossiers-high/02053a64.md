## SHA1 pitfall

**Primary mode:** Mode A — wrong pool word value (HIDDEN); `data_0219e474`, `data_0219e478`, `data_0219e480` are named globals whose addresses land in the literal pool; verify.py wildcards them; only ninja report catches a wrong address

**Why this recipe hits this mode:** `data_0219e474`, `data_0219e478`, and `data_0219e480` are named extern symbols loaded via LDR-from-pool. verify.py wildcards all pool-word relocations for named data symbols, so a wrong base address (e.g. off-by-4) passes verify.py silently and is only caught when `ninja report` resolves the relocations against the orig .s.

**Pool words to verify against orig .s:**
- `data_0219e474` — struct base for fields `f_0` (offset 0x00), `f_4c` (offset 0x4c), `f_50` (store target, offset 0x50); LDR base loaded twice (must reload each time, not bind to a register)
- `data_0219e478` — passed as `&data_0219e478` for the fifth stack argument to `func_02054ca8`
- `data_0219e480` — passed as `&data_0219e480` for the second argument to `func_02054ca8`

**Verify gate sequence:**
1. verify.py src/main/func_02053a64.c func_02053a64 [--cc all] — instruction bytes (pool words wildcarded)
2. ninja report — resolves pool words; catches Mode A global address mismatches for all three named symbols
3. ninja sha1 — final gate

## Reshape hints

- The two accesses to `data_0219e474.f_0` and `data_0219e474.f_4c` (and the store to `.f_50`) must each re-issue an LDR for the base address — do not bind the base to a local variable; mwcc must emit two separate LDR-pool loads (the risk note calls this out explicitly)
- Stack args to `func_02054ca8`: fifth arg `&data_0219e478` is `sp[0]`, sixth arg `8` is `sp[4]` — ensure these are the last two in the argument list and in that order

- `data_0219e478` and `data_0219e480` are `unsigned char` scalars; passing their addresses as `void *` is correct — do not change the types or add casts that move the address computation
- After `ninja report` confirms the three pool addresses match, the only remaining risk is the `f_50` store order relative to the two calls — it must come after `func_02055330()` returns, not interleaved
