# func_0200592c (main, HIGH class A, unknown B)

**Recipe:** straight call sequence + MMIO zero-store block (two bases)
**Risk:** Scheduling of zero-stores vs +0x1000 bridge may reorder; reg for #0 const.
**Confidence:** med

## SHA1 pitfall

**Primary mode:** Mode E — visible instruction byte diff from zero-store scheduling or register allocation for the #0 constant

**Why this recipe hits this mode:** mwcc may hoist the `r1=#0` or use different scratch registers for the two base pointers (`0x4000010` and `0x4001014`), causing a visible store-sequence or register diff; the +0x1000 bridge (`(char*)a + 0x1000`) may also merge or split into a different add sequence. All diffs are visible to `verify.py`.

**Pool words to verify against orig .s:**
No named-symbol pool words — raw hex constants only (0x4000010, 0x4001014 compared directly by verify.py).

**Verify gate sequence:**
1. verify.py src/main/func_0200592c.c func_0200592c [--cc all] — instruction diff is visible directly
2. ninja sha1 — final gate

## Reshape hints

- Write the zero-stores in exactly the order the orig .s emits `str`/`stm` instructions; do not combine into a `memset`-like loop — mwcc may optimize it
- The `+0x1000` bridge: `*(volatile int *)((char *)a + 0x1000) = 0` is required to force a separate literal; if mwcc merges it with `a[4]` offset, try casting through `uintptr_t` to prevent the fold
- If mwcc uses a different scratch register for `#0`: bind `const int zero = 0;` and use it for all five stores so the compiler reuses the same register rather than re-loading #0 from the constant pool
- The `b[0..2]` block at `0x4001014` is a separate base from `a`; keep the two pointer declarations separate and do not alias them — the compiler must emit two pool loads
- Verify the 11-call teardown order matches the orig .s BL sequence exactly before investigating the MMIO block; a misordered teardown call produces an early visible diff
